- RAG(Retrieval Augmented Generation)

  - 프라이빗 또는 독점적 데이터로 텍스트 생성을 보완하는 기술이다.
    - 대규모 데이터셋 또는 지식 기반(knowledge base)을 검색하기 위해 설계된 모델과 해당 모델을 통해 검색된 정보를 받아와 text 응답을 생성하는 대규모 언어 모델(LLM)과 같은 생성 모델을 결합한다.
    - 이를 통해 모델을 다시 훈련할 필요 없이 대규모 언어 모델의 출력을 향상시킬 수 있다. 
    - 지식 기반은 사람이나 시스템이 참고할 수 있도록 정리, 저장된 정보의 집합을 의미한다.
    - LLM 맥락에서 지식 기반은 주로 LLM이 직접 학습하지 않았거나, 최신이거나, 특정 도메인에 특화된 정보를 담고 있는 외부 저장소를 의미한다.
  - RAG의 동작 방식은 아래와 같다.
    - 사용자가 질의를 입력하고, 이 질의는 vector로 변환된다.
    - Embedding된 사용자의 질의와 가장 유사한 vector들을 조회한다.
    - 검색된 문서를 프롬프트에 포함하여 LLM에 전달한다.
    - LLM이 검색 결과를 근거로 응답을 생성한다.
  - Prompting
    - 생성형 AI에 input을 전달하는 과정이다.
    - Model의 output은 model에 얼마나 input을 잘 전달했는지에 따라 달라진다.
    - 이러한 작업을 prompt engineering이라고 부른다.
    - 더 자세한 설명은 [링크](https://www.promptingguide.ai/kr) 참조

  - Fine-tuning 방식
    - 이미 학습된 LLM의 가중치를 추가 데이터 학습을 통해 미세하게 조정하여 응답을 바꾸는 과정이다.
    - 지식을 조회하는 과정이 아니라 모델 내부 계산 결과의 확률 분포를 바꾸는 과정이다.
    - RAG와의 차이는 RAG는 검색한 결과를 prompt를 통해 LLM에 추가적인 지식을 주입하여 그 지식을 근거로 응답을 생성하는 방식인 반면, fine-tuning은 추가 지식을 LLM이 학습하여 계산 결과의 확률 분포를 바꾸는 방식이라는 것이다.
    - 즉, RAG의 경우 검색이 실행 되지만, fine-tuning의 경우 (결과적으로 검색에 사용되기는 하지만) 엄밀히 말해 검색은 아니다.
    - RAG는 광범위한 외부 지식을 검색하여 이를 텍스트 생성과 결합하는 방식인 반면, fine-tuning은 특정 목적을 위해 비교적 제한된 범위의 데이터에 집중한다.
    - 외부 자원에 접근할 수 있다는 점에서, RAG는 웹이나 기업 내부 지식 베이스로부터 실시간 또는 동적으로 변하는 정보를 활용해 응답을 생성해야 하는 작업에 특히 유용하다. 
    - 반면 fine-tuning은 수행해야 할 작업이 명확하게 정의되어 있고, 그 작업 하나에 대한 성능 최적화가 목표라면 fine-tuning은 매우 효율적인 방법이 될 수 있다. 
    - 두 기법 모두 각 작업마다 LLM을 처음부터 다시 학습시킬 필요가 없다는 공통적인 장점을 가진다.

  - RAG의 장점
    - Retrieval Augmented Generation(RAG)은 단독으로 동작하는 언어 모델에 비해 여러 가지 장점을 가진다.
    - RAG는 외부 참조 데이터를 정기적으로 갱신할 수 있기 때문에, 모델이 최신의 사실과 관련 정보를 항상 접근할 수 있도록 보장한다. 이를 통해 사용자의 질의와 관련될 수 있는 최신 정보를 반영한 응답을 생성할 수 있다.
    - RAG는 더 적은 연산 자원과 저장 공간을 필요로 하기 때문에 비용 효율적인 선택지이다. 자체적으로 LLM을 보유하거나, 모델을 fine-tuning하는 데 시간과 비용을 들일 필요가 없다.
    - RAG는 응답의 근거가 되는 외부 출처를 인용하여 사용자에게 제공할 수 있다. 사용자는 필요에 따라 해당 출처를 직접 확인함으로써, 자신이 받은 응답이 정확한지 검증할 수 있다.
    - LLM 기반 챗봇은 과거의 정형화된 응답 방식보다 개인화된 답변을 생성할 수 있지만, RAG는 그보다 한 단계 더 나아간 맞춤형 응답을 제공할 수 있다. 이는 RAG가 의도를 판단하는 과정에서 검색 기반 기법(주로 시맨틱 검색)을 활용하여, 문맥에 기반한 다양한 정보를 참조하면서 응답을 종합할 수 있기 때문이다.
    - 학습되지 않은 복잡한 질의에 직면했을 때, LLM은 때때로 부정확한 답변을 생성하는 ‘환각(hallucination)’ 현상을 보일 수 있지만 RAG는 관련 데이터 소스로부터 추가적인 참조 정보를 제공하여 응답을 근거에 기반해 생성함으로써, 모호한 질문에도 보다 정확하게 대응할 수 있다.
    - RAG 모델은 활용 범위가 넓어, 대화 시스템, 콘텐츠 생성, 정보 검색 등 다양한 자연어 처리 작업에 적용할 수 있다.
    - RAG는 검증된 외부 출처에 의존함으로써, 응답에 포함될 수 있는 편향(bias)을 완화하는 데 도움을 줄 수 있다.
  - RAG의 한계
    - RAG는 외부 지식에 의존하므로 검색된 정보가 부정확할 경우, 그에 기반해 생성되는 결과 역시 부정확해질 수 있다.
    - RAG의 검색(retrieval) 구성 요소는 대규모 지식 베이스나 웹을 대상으로 탐색을 수행하므로, 계산 비용이 크고 처리 속도가 느릴 수 있다. 다만 이는 여전히 fine-tuning에 비해서는 더 빠르고 비용도 적게 든다.
    - 검색과 생성 구성 요소를 매끄럽게 통합하기 위해서는 세심한 설계와 최적화가 필요하며, 이 과정에서 학습과 배포 단계 모두에서 잠재적인 어려움이 발생할 수 있다.
    - 외부 소스로부터 정보를 조회하는 과정은 민감한 데이터를 다룰 경우 개인정보 보호 이슈를 야기할 수 있다. 또한 개인정보 보호 및 컴플라이언스 요구 사항을 준수해야 하기 때문에, RAG가 접근할 수 있는 정보 소스가 제한될 수도 있다. 다만 문서 단위 접근 제어(document-level access)를 적용하여, 특정 역할(role)에 대해서만 접근 및 보안 권한을 부여하는 방식으로 이러한 문제를 해결할 수 있다.
    - RAG는 사실적 정확성에 기반한 접근 방식이기 때문에, 상상력이나 허구적 요소가 필요한 콘텐츠 생성에는 어려움을 겪을 수 있으며, 이로 인해 창작형 콘텐츠 생성 용도로의 활용에는 한계가 있다.







## Adaptive-RAG

> https://arxiv.org/html/2403.14403v2

- Adaptive-RAG

  - 개요

    - RAG는 매우 단순한 query부터 복잡한 query까지 다양한 query를 입력으로 받게 된다.
    - 때로는 매우 단순한 쿼리를 불필요할 정도로 과하게 처리하기도 하고, 때로는 복잡한 다단계(multi-step) query를 적절하게 처리하지 못 하기도 한다.
    - Adaptive-RAG는 상대적으로 작은 LM을 사용하여 query의 복잡도를 예측하고, 복잡도에 따라 검색 및 생성 전략을 동적으로 조정한다.

  - Single-Step Approach와 Multi-Step Approach

    - Single-step approach는 한 번의 검색으로 관련된 문서들을 찾은 후 이들을 통해 한 번만 답변을 생성하는 방식이다.
    - 이 방식은 multi-step reasoning을 필요로하는 복잡한 query에 대해 불충분한 답변을 반환할 수 있다는 문제가 있다.
    - 반면 multi-step approach는 검색한 결과를 통해 중간 답변을 생성하고, 중간 답변을 평가하여 query에 충분한 답변이 될 때까지 검색과 생성을 반복하는 방식이다.
    - 이 방식은 단순한 query에 대해서도 지나치게 복잡한 검색 및 생성 과정을 거치게 된다는 점에서 비효율적이라는 문제가 있다.

  - Adaptive-RAG

    - Query의 복잡도에 따라 다른 검색 및 생성 전략이 수행된다.
    - 매우 단순한 쿼리에 대해서는 검색을 건너뛰고 바로 답변을 생성하기도 하고, 중간 정도의 복잡도를 가진 쿼리는 single-step approach를 사용하며, 복잡한 쿼리에 대해서는 multi-step approach를 수행한다.

  - Multi-Step QA

    - 초기의 RAG는 주로 single-hop queries에 대한 답변을 생성하는데 집중했다.
    - 이런 종류의 쿼리들은 일반적으로 하나의 문서에서 답을 찾는 것이 가능했으며, 따라서 쿼리와 유사한 문서를 찾은 이후에 이들을 QA model에서 응답에 맞는 형식으로 재구성하는 방식으로 검색이 이루어졌다.

    - 그러나 하나의 문서로는 답을 찾을 수 없고, 여러 문서를 연결하거나 집계해서 답변해야 하는 쿼리들이 있다.
    - 또한 한 번의 검색으로는 답변을 생성할 수 없는 쿼리들 역시 존재하는데, 예를 들어 "When did the people who captured Malakoff come to the region where Philipsburg is located?"와 같은 쿼리는 여러 단계의 추론을 거쳐야한다.
    - 그러므로, 이런 복잡한 쿼리에 대한 답변을 생성하기 위해서 최근의 연구는 반복적인 검색과 답변 생성이 필요한 multi-step, multi-reasoning QA에 집중되고 있다.
    - 그러나 실제 세상에는 이렇게까지 복잡한 처리 과정이 필요하지는 않은 단순한 쿼리들도 많이 존재한다.



- 기존의 전략들
  - Non Retrieval
    - LLM을 $LLM$이라고 할때, $LLM$은 입력으로 연속된 token들 $x=[x_1, ..., x_n]$ 를 입력으로 받아 연속된 token들 $y=[y_1, ... , y_n]$을 생성하여 반한다.
    - 이를 수식으로 나타내면 $y=LLM(x)$와 같으며, 이를 QA의 관점에서 보면, input인  $x$는 쿼리인 $q$, output은 답변인 $\overline a$로 치환할 수 있다.
    - 치환하여 수식으로 나타내면 $\overline a = LLM(q)$와 같다.
    - 이상적으로는 답변 $\overline a$는 실제 정답 $a$와 같다.
    - 이 방식은 단순한 쿼리들을 처리하거나 방대한 양의 지식을 저장하는 LLM을 사용하는 경우에는 좋은 접근일 수도 있다.
    - 그러나 이 방식은 특정 인물, 사건 혹은 LLM의 지식을 넘어서는 주제에 대한 정확성과 최신성을 요구하는 쿼리에 대해서는 답변을 잘 하지 못 할수 있다는 문제가 있다.
  - Single Step Apporach
    - LLM만으로는 답변을 생성할 수 없는 쿼리들에 답변하기 위해서 외부 지식인 $d$를 사용할 수 있다.
    - $d$는 외부 지식 저장소 $D$에서 쿼리와 관련된 정보들을 수집해서 가지고 있다.
    - $D$로 부터 $d$를 얻기 위해는 query와 관련된 문서들 반환하는 특정한 retrieval model이 필요하다.
    - 외부 지식 저장소 $D$에서 외부 지식을 얻는 과정은 수식으로 $d=Retriever(q;D)$와 같이 표현할 수 있다.
    - Retrieval 단계가 완료되면 그 결과로 얻은 $d$를 포함하여 답변을 생성할 수 있는데, 이를 수식으로 $\overline a = LLM(q, d)$와 같이 표현할 수 있다.
    - $d$는 LLM이 가지고 있는 지식을 보완해줘, LLM이 생성하는 답변의 정확도와 최신성을 개선한다.
  - Multi Step Approach
    - Single step approach 역시 여전히 제한점을 가지고 있는데, 바로 여러 문서에 걸쳐있는 정보를 종합하고 이를 기반으로 추론해야 답변을 생성할 수 있는 복잡한 쿼리에 대해서는 충분한 답변을 제공하지 못한다는 점이다.
    - Multi step approach에서는 $LLM$이 $Retriever$과 여러 round에 걸쳐서 상호작용한다.
    - $LLM$은 최종 답변을 생성할때 까지 점진적으로 $q$에 대한 이해를 증진시킨다.
    - 매 단계 $i$마다 새로운 문서들인 $d_i$가 $D$로부터 수집된다.
    - 매 단계마다 새로워진 context $c_i$역시 $LLM$에 전달되는데, $c_i$는 이전 단계의 문서들($d_1, ..., d_{i-1}$)과 LLM이 생성한 답변들($\overline a_1, ..., \overline a_{i-1}$)로 구성된다.
    - 이를 수식으로 나타내면 $\overline a_i = LLM(q,d_i,c_i)$와 같이 나타낼 수 있다.
    - 또한 $Retriver$에도 context 정보를 함께 전달하며, 이를 수식으로 나타내면 $d_i=Retriever(q,c_i;D)$와 같이 나타낼 수 있다.
    - 이러한 방식은 LLM이 포괄적이고 광범위한 정보를 토대로 문서들을 연결시켜 보다 정확한 답변을 생성하게 하지만, 상당한 resource를 필요로 한다.



- Adaptive RAG(Adaptive Retrieval-Augmented Generation)
  - Query 복잡도 평가
    - 주어진 쿼리의 복잡도를 평가하여 복잡도 수준을 반환하는 complexity classifier model을 사용하여 쿼리의 복잡도를 평가한다.
    - Classifier는 수식으로 $o=Claasifier(q)$와 같이 표현할 수 있다.
    - $Classifier$는 쿼리의 복잡도를 분류하기 위해 학습된 smaller Language Model이며, $o$는 세 단계로 구분되는 복잡도 수준이다.
    - 복잡도는 A, B, C 세 단계로 구분되며, A는 LLM만으로 답변이 가능한 복잡도, B는 single-step approach로 답변이 가능한 복잡도, C는 multi-step approach로 답변이 가능한 복잡도를 가리킨다.
  - 학습 전략
    - Classifier로 사용할 smaller Language Model을 학습시킬 dataset을 구축하기 위해서 Labeling 전략과 benchmark dataset 전략을 사용했다.
    - 이렇게 구축된 dataset을 사용하여 cross-entropy loss로 Classifier를 학습시킨다.
  - Labeling 전략
    - 세 가지 서로 다른 RAG LLM 전략(non retrieval, single step, multi step)의 결과를 바탕으로 쿼리의 복잡도에 레이블을 부여하여, 해당 쿼리의 필요에 따라 레이블을 결정하고자 한다. 
    - 예를 들어, 가장 단순한 비검색 기반 방식이 정답을 올바르게 생성한다면, 해당 쿼리의 레이블은 'A'로 지정된다. 
    - 또한, 쿼리에 레이블을 부여할 때 서로 다른 모델 간의 동점을 해소하기 위해, 더 단순한 모델에 높은 우선순위를 부여한다. 
    - 즉, 비검색 기반 방식은 실패하지만 단일 단계 방식과 다단계 방식 모두 동일하게 정답을 생성하는 경우, 해당 쿼리에는 레이블 'B'를 부여한다.
    - 그러나 이러한 레이블링 전략에는 한계가 있는데, 세 가지 검색 증강 방식이 모두 정답을 생성하지 못하는 경우 일부 쿼리에는 레이블이 부여되지 않을 수 있다는 점이다.
  - Benchmark datasets 전략
    - Benchmark dataset의 생성 방식을 고려할 때, 어떤 전략이 잘 맞는지가 이미 어느 정도 정해져 있다고 볼 수 있다.
    - 예를 들어 순차적 추론을 요구하는 QA 데이터셋은 일반적으로 다단계 방식을 필요로 하는 반면, 단일 문서가 레이블로 지정된 데이터셋의 쿼리는 단일 단계 방식으로 이상적으로 답변 가능하다.
    - 따라서 첫 번째 레이블링 단계 이후에도 레이블이 부여되지 않은 쿼리에 대해서는, 단일 홉 데이터셋의 쿼리에는 'B'를, 다중 홉 데이터셋의 쿼리에는 'C'를 부여한다.



- 결과와 한계
  - 결과
    - Adaptive-RAG는 다양한 쿼리 복잡도에 대해 지나치게 단순하거나 지나치게 복잡한 방식을 획일적으로 적용하는 기존의 단일 접근 방식들과 비교하여, 복잡한 쿼리 처리에는 더 많은 자원을 할당하고 단순한 쿼리는 효율적으로 처리함으로써 QA 시스템의 전반적인 정확도와 효율성을 향상시키는 것으로 나타났다.
  - 한계
    - 학습 데이터셋과 아키텍처 관점에서 분류기를 개선할 수 있는 잠재적 여지가 존재한다. 
    - 구체적으로, 쿼리 복잡도 분류기를 학습시키기 위한 데이터셋이 존재하지 않기 때문에, 모델 예측 결과와 데이터셋의 귀납적 편향을 바탕으로 새로운 데이터를 자동으로 생성했으나 레이블링 과정은 쿼리 복잡도를 레이블링하는 하나의 특정 구현 방식이며, 그 효과성에도 불구하고 일부 쿼리를 잘못 레이블링할 가능성이 있다. 
    - smaller Language Model에 기반한 분류기 설계는 쿼리 복잡도를 분류하기 위한 초기의 가장 단순한 구현이며, 이를 토대로 향후 연구에서 분류기 아키텍처와 성능을 개선한다면 전반적인 QA 성능 향상에도 긍정적으로 기여할 것이다.
